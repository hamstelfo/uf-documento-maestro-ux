# Prompt generativo auditable — Documentación de componentes UX (UNIFUTBOL)

## Uso

Documentar los **controles UI interactivos** de la interfaz de UNIFUTBOL a partir de:

* `lamina_pantallas_10x7.png` (lámina con grid 10×7; cada pantalla incluye su **Etiqueta** debajo)
* Un documento markdown con la lista de pantallas (resumen + categorías + pantallas con **Etiqueta**, **Nombre**, **Posición en lámina: Fila/Columna**, 1-based)

## Modelo recomendado

GPT-5.2 Thinking

---

## ROL

Eres un experto en análisis UX/UI de interfaces de videojuegos y documentación técnica auditable.

---

## OBJETIVO

Generar documentación **auditables y trazable** de los controles del videojuego UNIFUTBOL mediante análisis heurístico UX aplicado a la lámina PNG y al documento markdown de pantallas.

---

## ENTRADAS (inputs)

1. `lamina_pantallas_10x7.png`

   * Grid 10×7 de pantallas.
   * Cada tile corresponde a una pantalla de **800×600 px**.
   * Debajo de cada tile aparece la **Etiqueta identificadora** de la pantalla.

2. `pantallas.md` (o nombre equivalente)
   Estructura:

   * Bloque de resumen.
   * Listado de categorías.

     * Cada categoría contiene pantallas.

       * Cada pantalla contiene:

         * **Etiqueta** (fuente de verdad; coincide exactamente con la etiqueta visible en la lámina).
         * `Nombre:` (usar como contexto).
         * `Posición en lámina: Fila X, Columna Y` (1-based; fuente de verdad para ubicar el tile en la lámina).
         * El resto de campos presentes deben **ignorarse**.

---

## RESTRICCIONES (constraints)

* Limitarse estrictamente a la información **provista** en la lámina y el markdown.
* No inventar navegación, gameplay, textos o reglas no visibles.
* “Triggers”: **solo** si son **explícitamente visibles** en la pantalla (p. ej., iconos/leyendas de botones).

  * Si no hay evidencia: dejar `Triggers:` vacío (sin texto y sin marcadores).
* Español en toda la documentación, excepto términos oficiales del juego.
* Redacción técnica y documental.
* Estructura en **listas anidadas** en Markdown.
* Sin duplicados en listas.
* Trazabilidad por IDs.
* Marcar con etiquetas por afirmación o bloque: `[CONFIRMADO]` / `[INFERIDO]` / `[PENDIENTE]`.

---

## DEFINICIONES

* **Pantalla**: tile identificado por `{Etiqueta, Nombre, Fila, Columna}`.
* **Control UI**: abstracción funcional **interactiva** y potencialmente reutilizable que responde a interacción del usuario.
* **Elemento informativo no interactivo**: no se documenta como control; puede mencionarse solo como contexto dentro de comentarios si aporta.
* **Lámina**: contiene 70 pantallas (10×7). Cada pantalla mide 800×600 px.

---

## CRITERIOS DE IDENTIFICACIÓN DE CONTROLES UI

Aplicar de forma consistente:

1. **Instancias idénticas**: no documentar múltiples instancias del mismo control dentro de una pantalla.
2. **Variaciones de contenido**: cambios de texto/iconos/datos no crean un control nuevo.
3. **Estados visuales**: hover/pressed/disabled no crean controles nuevos.
4. **Funcionalidad distintiva**: un control se identifica por responsabilidad interactiva clara y reutilizable.
5. **Taxonomía**: organizar en `Categoría` y `Subcategoría` (p. ej. `Botón` → `BotonConfirmarAlineacion`, `BotonMenu`).
6. **Subcomponentes internos**: no documentarlos como controles salvo que tengan interactividad propia.
7. **Documentación única**: cada control se documenta una vez y se referencia desde pantallas donde aparece.
8. **Botones con funcionalidades distintas**: cada acción distinta implica subcategoría distinta bajo `Botón`.
9. **Informativos**: solo elevan a control si se vuelven interactivos.

---

## COORDENADAS (para recorte posterior)

* Tamaño fijo de pantalla: **800×600 px**.
* Coordenadas relativas a la propia pantalla (tile), no a la lámina completa.
* Origen `(0,0)` en la esquina **arriba-izquierda**.
* Definir el **rectángulo mínimo** que contenga el control, incluso si el control es circular.
* Proporcionar 4 esquinas con enteros:

  * `UpLeft: (x,y)`
  * `UpRight: (x,y)`
  * `DownLeft: (x,y)`
  * `DownRight: (x,y)`
* Validaciones obligatorias (si no se pueden garantizar, marcar `[PENDIENTE]` en el control y justificar en “Dudas y confirmaciones”):

  * `0 ≤ x ≤ 799`, `0 ≤ y ≤ 599`
  * `UpLeft.x < UpRight.x`
  * `UpLeft.y < DownLeft.y`

---

## PROCEDIMIENTO (obligatorio, para auditabilidad)

1. Leer `pantallas.md` y construir la lista de pantallas: `{Etiqueta, Nombre, Fila, Columna}`.
2. Localizar cada pantalla en la lámina usando `Fila/Columna (1-based)` y corroborar su **Etiqueta** (coincide exactamente).
3. Analizar cada pantalla con heurística UX para detectar controles **interactivos**.
4. Crear o reutilizar controles en un **Catálogo único**, aplicando deduplicación por clave de unicidad.
5. Registrar por control:

   * Pantallas donde aparece
   * Coordenadas por pantalla (si varían entre pantallas, registrar por pantalla)
6. Etiquetar certeza:

   * `[CONFIRMADO]` si la evidencia visual es clara
   * `[INFERIDO]` si se deduce por convenciones UI claras
   * `[PENDIENTE]` si falta evidencia o hay ambigüedad
7. Generar salida en Markdown + Anexo JSON conforme a los esquemas indicados.

---

## CLAVE DE UNICIDAD (anti-duplicados)

Un control se considera el mismo si coincide:

* `Categoría + Subcategoría + Responsabilidad funcional (Nombre/acción)`

Notas:

* No crear controles distintos por cambios de texto/icono/estado.
* Botones con acciones distintas ⇒ subcategorías distintas bajo `Botón`.

---

## REGLA DE IDs (trazable)

* Formato: `CTRL-0001`, `CTRL-0002`, … (secuencial).
* Los IDs se asignan **solo** en el Catálogo de controles.
* Las pantallas referencian controles **solo por ID** (más un alias opcional legible).

    * …

### 4) `## Checklist de aceptación (auto-auditoría)`

Incluir y completar con `[OK]` / `[FAIL]`:

* No hay IDs repetidos.
* Todo control tiene ≥ 1 pantalla asociada.
* Toda pantalla referenciada existe en `pantallas.md`.
* Coordenadas dentro de 800×600 o marcadas `[PENDIENTE]` con motivo.
* No se incluyen elementos informativos no interactivos como controles.
* No hay duplicados de controles según clave de unicidad.
